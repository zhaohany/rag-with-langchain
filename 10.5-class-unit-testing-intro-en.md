# Unit Testing Intro

## What Is a Unit Test

A Unit Test verifies whether a small functional unit behaves as expected. In most cases, that unit is a function.

In this project, `split_into_chunks` in `app/shared/chunking.py` is a good beginner-friendly testing target because:

- The input and output are clear.
- The behavior is repeatable and stable, meaning deterministic.
- It does not depend on external services, so failures are easier to locate.

## Why Write Unit Tests

- After code changes, tests can catch regression issues early.
- Tests turn “expected behavior” into executable documentation.
- Tests give more confidence during refactoring and reduce the chance of breaking existing behavior.
- When a test fails, it helps locate the specific logic that is wrong.

## Tools Used in This Project

- `pytest`: Python testing framework.
- `make test`: runs the full test suite. The underlying command is `python3 -m pytest -q`.

`pytest` is already included in `requirements.txt`.

Common commands:

```bash
make test
python3 -m pytest tests/test_chunking.py -q
python3 -m pytest tests/test_chunking.py::test_split_into_chunks_basic_split -q
```

## Pytest Basics

### 1) Test Naming Convention

- File names start with `test_`.
- Test function names start with `test_`.

Example:

```python
def test_split_into_chunks_basic_split() -> None:
    ...
```

### 2) Normal Assertions (Value Assertions)

Use `assert` to verify results:

```python
assert chunks
assert len(chunks) == 3
assert all(len(chunk) <= 45 for chunk in chunks)
```

Explanation:

- `assert chunks`: asserts that the list is not empty. An empty list is evaluated as `False`.
- `assert len(chunks) == 3`: asserts that the number of chunks is exactly 3.
- `assert all(...)`: asserts that every element satisfies the condition. If any element does not, the test fails.

Recommendations:

- Write “structure assertions” first, such as non-empty and count checks, then write “content assertions”, such as exact values and formats.
- Failure messages should help locate the issue. Add a message string when needed:

```python
assert len(chunks) == 3, "expected exactly 3 chunks"
```

### 3) Exception Assertions

Use `pytest.raises` to verify exception branches:

```python
with pytest.raises(ValueError, match="chunk_size"):
    split_into_chunks(text="hello", chunk_size=0, chunk_overlap=0)
```

Explanation:

- `ValueError`: the expected exception type.
- `match="chunk_size"`: the expected exception message contains `chunk_size`. This uses regex matching.
- The call inside the `with ...:` block must raise this exception, otherwise the test fails.

Common mistakes:

- Only writing `pytest.raises(Exception)`: too broad and not helpful for debugging.
- Not writing `match`: the code may raise an error for the wrong reason, but the test still passes.

### 4) Common Variable Meanings (Using Chunking Tests as an Example)

```python
text = "# Title\n\nBody"
chunk_size = 20
chunk_overlap = 4
chunks = split_into_chunks(text=text, chunk_size=chunk_size, chunk_overlap=chunk_overlap)
```

- `text`: the original input text to be split.
- `chunk_size`: the maximum length of a single chunk, measured in characters.
- `chunk_overlap`: the number of overlapping characters between two neighboring chunks.
- `chunks`: the function return value, with type `list[str]`.

Properties worth checking first:

- `len(chunks)`: how many chunks are produced after splitting.
- `len(chunks[i]) <= chunk_size`: whether each chunk length is valid.
- Whether adjacent chunks overlap: `chunks[i][-chunk_overlap:] == chunks[i+1][:chunk_overlap]`, when applicable.

### 5) Variable Naming Recommendations

- Use semantic names. Avoid `a`, `b`, and `x`.
- In tests, variable names should directly express the business meaning, for example:
  - `input_text`
  - `expected_chunks`
  - `actual_chunks`
  - `invalid_chunk_size`

Example:

```python
input_text = "hello"
invalid_chunk_size = 0

with pytest.raises(ValueError, match="chunk_size"):
    split_into_chunks(text=input_text, chunk_size=invalid_chunk_size, chunk_overlap=0)
```

## Recommended Style: AAA Pattern

- Arrange: prepare input data.
- Act: call the function under test.
- Assert: assert the result or exception.

Example structure:

```python
def test_example() -> None:
    # Arrange
    text = "# Title\n\nBody"

    # Act
    chunks = split_into_chunks(text=text, chunk_size=20, chunk_overlap=0)

    # Assert
    assert chunks
```

## How to Teach Someone Writing Tests for the First Time

Use this flow:

1. Run `make test` once first.
2. Read one failure message completely.
3. Open the corresponding test function.
4. Add one assertion first. Do not write too much at once.
5. Re-run only this test function.
6. After it passes, move to the next test.

Core principle: focus on one behavior at a time, and fix one failure at a time.
